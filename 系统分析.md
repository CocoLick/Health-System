第三章 系统分析

本章围绕个性化营养膳食智能规划系统展开全方位的功能性与静态结构分析，通过用例图明确不同角色的功能权限与用例关联，结合用例规格说明和活动图梳理各功能的执行流程与业务逻辑，同时借助分析类图呈现系统核心领域类的属性及类间固有数据关联，为系统后续设计与开发奠定核心分析基础。

3.1 功能性需求分析

3.1.1 用例图

![img](file:///C:\Users\蔡霆峰\AppData\Local\Temp\ksohtml19520\wps7.jpg) 

 

 

本系统用例图以普通用户、规划师、系统管理员三类角色为核心参与者，构建了覆盖系统全业务的用例体系，不同角色基于自身业务定位拥有差异化的用例操作权限，且存在部分用例为多角色共用的业务特征，从角色视角的用例分布及关联关系如下：

（1）普通用户：作为系统的核心使用主体，围绕个人健康管理全流程拥有操作权限，可执行账号管理、个人信息与健康数据管理、营养摄入监测与反馈、膳食计划管理、食材管理、智能膳食推荐、意见与反馈管理、统计数据管理用例。其中账号管理为基础操作用例，支撑用户的系统登录与账号维护；个人信息与健康数据管理、营养摄入监测与反馈为基础数据采集用例，为后续膳食相关服务提供数据支撑；膳食计划管理、食材管理、智能膳食推荐为核心膳食服务用例，是用户健康管理的核心功能落地；意见与反馈管理、统计数据管理为辅助服务用例，分别支撑用户的系统互动与个人数据记录，各用例按 “基础操作 - 数据采集 - 核心服务 - 辅助服务” 的业务逻辑形成递进关联，共同实现普通用户的个人健康膳食管理需求。

（2）规划师：作为系统的专业服务主体，围绕为普通用户提供专业膳食指导设计用例权限，可执行账号管理、个人信息与健康数据管理、营养摄入监测与反馈、膳食计划管理、健康教育管理、统计数据管理、意见与反馈管理用例。其中账号管理为基础操作用例，保障规划师的系统登录与账号维护；个人信息与健康数据管理、营养摄入监测与反馈为数据查看用例，支撑规划师获取用户健康及饮食数据，是专业服务的前提；膳食计划管理、健康教育管理为核心专业服务用例，分别实现规划师的膳食计划制定与健康知识科普；统计数据管理、意见与反馈管理为辅助服务用例，支撑规划师查看服务相关数据、处理用户咨询反馈，各用例按 “基础操作 - 数据获取 - 核心服务 - 辅助服务” 的业务逻辑关联，形成规划师的专业服务闭环。

（3）系统管理员：作为系统的运维管理主体，围绕系统运营与内容审核设计最小必要用例权限，可执行账号管理、膳食计划管理、健康教育管理、统计数据管理、意见与反馈管理用例。其中账号管理为基础运维用例，支撑管理员对全系统用户账号的权限管理与状态维护；膳食计划管理、健康教育管理为核心审核用例，是管理员对规划师发布的专业内容进行科学性、合规性审核的核心功能；统计数据管理、意见与反馈管理为系统运营用例，分别支撑管理员查看系统整体运营数据、处理用户的核心反馈诉求，各用例按 “基础运维 - 内容审核 - 运营管理” 的业务逻辑关联，保障系统的规范运营与稳定运行。

此外，账号管理、膳食计划管理、统计数据管理、意见与反馈管理为三类角色的共用用例，不同角色对共用用例的操作权限存在差异化：账号管理中，普通用户与规划师仅可操作自身账号，管理员可管理全系统账号；膳食计划管理中，普通用户仅可查看与执行计划，规划师可制定与修改计划，管理员可审核计划；意见与反馈管理中，普通用户仅可提交反馈，规划师、管理员可回复反馈，其余用例为单一角色专属用例，实现角色间的功能边界划分，保障系统业务的有序开展。

3.2 系统静态结构分析（分析类图）

![img](file:///C:\Users\蔡霆峰\AppData\Local\Temp\ksohtml19520\wps17.jpg) 

本系统分析类图为系统静态结构分析的核心模型，聚焦业务数据的本质关联，共设计用户、规划师、膳食计划、健康数据、营养记录、食材、健康教育、意见反馈8 个核心领域类，类与类之间体现数据的归属、创建、包含等固有业务逻辑，具体说明如下：

（1）核心领域类及属性

用户类：系统核心数据实体，为普通用户的数字化映射，包含用户 ID、用户名、登录密码、手机号码、性别、年龄、邮箱、账号创建时间属性，其中用户 ID 为唯一标识，其余属性完整记录用户的基础身份信息，是所有个人业务数据的归属主体，支撑系统对个人用户的精准管理。

规划师类：系统专业服务数据实体，为规划师的数字化映射，包含规划师 ID、姓名、登录密码、职称、专业领域、联系方式、账号创建时间属性，其中规划师 ID 为唯一标识，职称、专业领域等属性体现其专业特征。

膳食计划类：系统核心业务数据实体，包含计划 ID、用户 ID、规划师 ID、计划标题、饮食目标、计划内容详情、审核状态、发布时间、最近更新时间属性，计划 ID 为唯一标识，用户 ID、规划师 ID 分别关联归属用户与创建规划师，审核状态标记计划的审核结果，发布与更新时间记录计划的生命周期信息，完整呈现膳食计划的核心特征。

健康数据类：用户健康基础数据实体，包含数据 ID、用户 ID、身高、体重、血压值、血糖值、心率、过敏病史、数据录入时间属性，数据 ID 为唯一标识，用户 ID 关联归属用户，身高、血压值等属性记录用户核心健康指标，过敏病史为膳食计划制定提供重要参考，录入时间标记数据的采集节点。

营养记录类：用户饮食营养数据实体，包含记录 ID、用户 ID、摄入日期、摄入时间段、食物名称、摄入热量、蛋白质、碳水化合物、脂肪、纤维素、记录录入时间属性，记录 ID 为唯一标识，用户 ID 关联归属用户，摄入日期、时间段实现饮食记录的时间精准化，各类营养素指标完整统计用户的饮食营养摄入情况。

食材类：膳食计划基础数据实体，包含食材 ID、食材名称、食材类别、营养成分详情、计量单位、每 100 克热量属性，食材 ID 为唯一标识，食材类别、营养成分详情等属性记录食材的核心特征与营养指标，是膳食计划制定的基础数据支撑。

健康教育类：系统专业知识数据实体，包含内容 ID、规划师 ID、文章标题、健康分类、图文内容详情、审核状态、发布时间、最近更新时间属性，内容 ID 为唯一标识，规划师 ID 关联创建规划师，健康分类实现内容的精准归类，审核状态标记内容的审核结果，完整呈现健康教育内容的核心信息。

意见反馈类：系统互动数据实体，包含反馈 ID、用户 ID、反馈标题、反馈内容、反馈类型、处理状态、反馈时间属性，反馈 ID 为唯一标识，用户 ID 关联提交用户，反馈类型、处理状态分别标记反馈的类别与处理进度，完整记录用户反馈的全流程信息。

（2）类与类之间的关联关系

所有类间关联均为1 对多（1-n） ，具体如下：

用户与膳食计划：1-n 关联，关系描述为 “拥有”，即一个用户可拥有多个膳食计划，适配用户在不同健康阶段、不同饮食目标下的膳食需求，一个膳食计划仅唯一归属一个用户。

用户与健康数据：1-n 关联，关系描述为 “录入”，即一个用户可录入多条不同时间的健康数据，形成个人健康数据档案，一条健康数据仅唯一归属一个用户。

用户与营养记录：1-n 关联，关系描述为 “记录”，即一个用户可记录多条不同日期、时段的营养摄入信息，形成个人饮食营养档案，一条营养记录仅唯一归属一个用户。

用户与意见反馈：1-n 关联，关系描述为 “提交”，即一个用户可提交多条不同类型的意见与反馈，一条意见反馈仅由一个用户提交。

规划师与膳食计划：1-n 关联，关系描述为 “制定”，即一个规划师可为多个用户制定个性化膳食计划，一个膳食计划仅由一个规划师创建。

规划师与健康教育：1-n 关联，关系描述为 “发布”，即一个规划师可发布多条不同类别的健康教育内容，一条健康教育内容仅由一个规划师创建。

规划师与意见反馈：1-n 关联，关系描述为 “回复”，即一个规划师可回复多个用户的反馈，一条意见反馈仅由一个规划师处理回复。

膳食计划与食材：1-n 关联，关系描述为 “包含”，即一个膳食计划可包含多种不同的食材，实现膳食搭配的多样性，食材作为基础数据，可被多个膳食计划引用。

整体分析类图通过 8 个核心领域类及对应的 1-n 关联关系，精准呈现了系统的核心数据结构与业务数据之间的固有归属逻辑，为后续系统的数据库设计、功能模块开发提供了清晰、规范的概念模型支撑，保障了系统设计与开发的一致性和合理性。
